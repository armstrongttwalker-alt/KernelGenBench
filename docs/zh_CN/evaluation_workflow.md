# FlagBench 算子评测流程

本文档介绍 FlagBench 如何对 LLM 生成的 Triton kernel 进行评测。

---

## 1. 评测目标

FlagBench 的核心目标是评估 LLM 生成 Triton kernel 的能力。给定一个算子规范（如 `torch.add` 的函数签名和语义描述），LLM 需要生成对应的 Triton 实现，系统自动验证其正确性。

---

## 2. 整体流程

```
┌───────────────────────────────────────────────────────────────┐
│                      Pass@K 评测循环                           │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│    算子规范  ──▶  LLM 生成  ──▶  并行验证  ──▶  统计结果       │
│                     │              │                          │
│                     │              ▼                          │
│                     │         Pass/Fail                       │
│                     │              │                          │
│                     ◀──────────────┘                          │
│                  (失败反馈，下一轮重试)                         │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

---

## 3. Pass@K 评测方法

我们采用 **Pass@K** 作为核心评测指标，衡量 LLM 在 K 次尝试内生成正确实现的概率。

### 3.1 多轮迭代

- **Round 0**：对所有算子生成代码并验证，记录通过的算子
- **Round 1**：仅对未通过的算子重新生成并验证
- **Round 2 ~ K-1**：继续迭代，直到全部通过或达到最大轮次

每轮只处理**尚未通过**的算子，已通过的不再重复测试。

### 3.2 Reflection 机制

系统支持 **Reflection（反思）** 模式：将上一轮的失败信息（错误类型、失败用例、错误堆栈）反馈给 LLM，引导其在下一轮修正代码，提升生成质量。

### 3.3 评测指标

- **Pass@1**：首次生成即通过的比例
- **Pass@K**：K 轮内通过的累计比例

---

## 4. 验证机制

### 4.1 并行验证

为加速评测，系统将测试任务分配到多个 GPU 并行执行，每个测试运行在独立进程中，设有超时保护机制。

### 4.2 正确性判定

1. 将生成的 Triton kernel 注册到 PyTorch dispatch
2. 执行参数化测试（覆盖不同 shape、dtype 组合）
3. 与 PyTorch 原生实现的输出对比
4. 在容差范围内判定通过

单个算子通常有数百个测试用例，覆盖各种边界条件。

---

## 5. 两类算子的处理

### 5.1 PyTorch 算子

如 `aten::add`、`aten::mm` 等，直接以 PyTorch 原生实现作为参考基准。

### 5.2 非 PyTorch 算子

如 cuBLAS 中的 `sgemm`、`dgemm` 等，需要预先提供 baseline 实现作为参考基准。

---

## 6. 评测输出

每轮评测结束后，系统记录：

- 本轮新通过的算子列表
- 累计通过率（Pass@K）
- 失败算子的错误信息，用于后续 Reflection
